# 疾病领域调研（InnoSight 专用完整版）

## 目标

以医药行业高级分析师的标准，优先依靠 InnoSight 的结构化医药数据建立完整母表，再用监管、试验注册、指南、论文和交易双方公告核验关键结论。覆盖率、可追溯性和决策价值优先于响应速度与篇幅节省。

## 不可妥协的运行条件

- 必须检测到 `innosight_mcp`，并首先调用 `mcp__innosight_mcp__list_data_apis`。
- 调用任何数据 API 前，必须以目标 API path 调用 `mcp__innosight_mcp__get_data_api_schema`；不得猜测字段、枚举或分页参数。
- 优先调用专用数据接口；只有专用接口不适用时才使用 `mcp__innosight_mcp__query_data_api`。
- 对管线、临床试验、交易、新闻等集合型结果完整翻页，不得只取第一页或任意截断代表性结果。
- InnoSight 可用时，不得先以网页搜索代替数据库母表；外部来源用于补缺和一手核验。
- 若 InnoSight 未启用、无权限或核心接口持续失败，停止假装执行完整版，明确说明阻塞，并建议改用 `disease-area-research` 通用版。

## 输入与默认范围

- 将疾病名称作为唯一必填输入。若用户未提供，只追问具体疾病领域。
- 默认覆盖全球与中国、创新治疗药物及主要亚型；数据截止日为执行当天。
- 默认回溯最近 24 个月 BD 与并购。
- 对宽泛疾病自动建立亚型层级和纳入/排除边界；除非疾病本身无法识别，否则不要求用户重复指定报告结构。
- 默认使用中文和专业、客观、审慎的医药行业分析口吻。

## 执行前必须读取

完整读取以下文件后再调用数据：

1. `references/source-and-research-protocol.md`：精确工具顺序、接口覆盖、分页、外部核验与冲突处理。
2. `references/golden-schema.md`：完整五部分报告结构、字段和分析要求。
3. `references/quality-checklist.md`：交付前覆盖率与证据检查。

## 执行流程

### 0. InnoSight 预检

1. 调用 `mcp__innosight_mcp__list_data_apis`，保存可用接口清单。
2. 将接口清单映射到流行病学、指南/治疗格局、已获批药物、全球管线、临床试验设计与时间线、试验结果、中国注册、BD交易、新闻和专利背景。
3. 对计划使用的每个接口读取 schema，记录 API path、准确参数、分页方式和返回字段。
4. 若关键数据域不存在，先记录缺口，再规划官方外部来源补缺；不得直接跳过。

### 1. 疾病范围与标准疗法

1. 规范中文名、英文名、缩写、历史名称、主要亚型和检索同义词。
2. 获取全球与中国流行病学、疾病负担、最新指南和现行 SoC。
3. 从患者结局、疗效、安全性、复发/耐药、特定人群、诊断、给药、可及性与成本形成未满足需求清单。

### 2. 全球管线母表

1. 使用疾病总称、同义词、旧称和关键亚型分别检索并完整翻页。
2. 按资产别名、盐型、合作方和地区权益归一去重。
3. 对全部活跃资产记录靶点/机制、modality、企业、全球最高阶段、中国阶段、目标人群、试验和最新状态。
4. 按靶点/机制和 modality 双维度统计阶段、拥挤度和主要企业。
5. 对已上市、关键/注册性试验、Phase 2/3、重要读出或机制差异资产继续查询设计、时间线和结果。

### 3. 中国本土布局

1. 从全球母表中识别总部或主要创新研发主体位于中国的企业。
2. 区分自主发现、License-in、共同开发和区域权益。
3. 核对临床前、IND、Phase 1/2/3、NDA/BLA、获批上市和商业化状态。
4. 分析同质化、潜在 FIC/BIC、本地可及性、数据质量、开发、IP 和商业化风险。

### 4. BD 与并购

1. 系统检索截止日前 24 个月的许可、合作研发、共同商业化、选择权、资产收购和公司并购。
2. 对疾病总称、主要亚型、核心靶点和重点资产补检。
3. 记录日期、双方、标的、机制、权益地区、首付款、股权/近期付款、里程碑、潜在总额、royalty 和状态。
4. 未披露写“未披露”；不得重复加总或将 `up to` 写成确定付款。

### 5. 外部一手核验

对关键数字、阶段、审批、临床结果、指南和交易金额，使用 FDA、NMPA/CDE、EMA、ClinicalTrials.gov、ChiCTR、论文、会议、监管申报或交易双方公告核验。记录发布日期、数据截止日和访问日期。

### 6. 撰写与质控

1. 严格使用 Golden Schema 的五个一级部分，不增加平行的第六部分。
2. 重要管线、临床结果和交易使用结构化表格；表后给出判断而非条目堆积。
3. 区分已核验事实、跨来源推断、分析判断和未知事项。
4. 完成质量检查后，先输出简短“研究执行回执”，再输出正式报告。

## 研究执行回执

正式报告前必须给出紧凑回执，至少包含：

| 数据域 | InnoSight API/path | Schema 已读取 | 查询词/过滤条件 | 页数或 cursor 批次 | 返回/去重数量 | 外部核验 | 状态/缺口 |
|---|---|---|---|---:|---:|---|---|

回执规则：

- 不泄露账号凭证、内部 token 或敏感配置。
- 无法获得准确页数时，写明分页终止依据，例如 `has_more=false`、无 `next_cursor` 或已达到 `totalCount`。
- 任何核心数据域未完成时标记“未完成”，并在报告相应章节降低结论置信度。
- 不得用“已调用 InnoSight”一句话代替接口、分页和结果数量证明。

## 质量底线

- 不把搜索摘要、模型常识或新闻稿作为关键数字的唯一依据。
- 不把试验获批写成实际启动，不混淆全球阶段与中国阶段。
- 不把单臂早期数据写成确证性比较证据，不对不可比跨试验数字作简单排名。
- 不混用患病、确诊、治疗、可治疗和市场患者池。
- 不凭企业宣称判定 FIC/BIC。
- 不把专利背景检索表述为正式 FTO 或法律意见。
- 对检索不到、未披露、接口缺失或来源冲突的内容明确标注，不补写数字。
